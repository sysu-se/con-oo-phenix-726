# 设计说明

## 1. 核心问题

### 1.1 Svelte 响应式机制与领域对象协作方式

本实现采用 Store Adapter 方案即 A 方案作为响应式边界。`Sudoku` 与 `Game` 保持为纯业务对象，不直接承担 Svelte 订阅逻辑；适配层在每次领域操作后将最新快照写入 `writable` store，组件通过 `$store` 自动订阅并重渲染。

协作过程如下：

1. 组件触发用户行为（输入、撤销、重做、提示）。
2. 适配层调用 `Game` / `Sudoku` 的领域方法完成状态演进。
3. 适配层调用同步函数，将 `game.toJSON().sudoku` 快照写入响应式 store。
4. Svelte 根据 store 变化刷新界面。

此机制保证了领域规则与 UI 刷新之间的清晰分工：领域层负责使状态正确，Svelte 负责状态的展示。
本实现依赖的是“写入 store 新值触发刷新”，而不是通过直接修改对象内部字段来触发刷新。

### 1.2 View 层消费 `Sudoku` / `Game` 的方式

View 层不直接持有可变领域对象，而是消费（即使用） `@sudoku/stores/grid` 暴露的适配层接口。  
适配层内部持有 `Game`，并由 `Game` 持有 `Sudoku`。

View 层消费内容可分为两类（第3部分有详细说明）： 

- 响应式状态：用于页面显示的数据，例如盘面、冲突格、撤销/重做按钮是否可用，响应式状态一旦数据变化，界面会自动跟着更新。
- 操作命令：用于执行操作的方法，例如输入数字、撤销、重做。组件在用户点击或按键时调用这些方法，方法再把操作交给领域层处理。

此设计确保了 UI 只能通过领域入口驱动状态变化，避免绕过历史管理和规则校验。

## 2. 领域对象设计与职责

### 2.1 `Sudoku` 的职责

`Sudoku` 负责盘面数据与规则约束，核心职责如下：

- 持有初始盘面 `initial` 与当前盘面 `cells`
- 提供 `guess(move)` 作为唯一落子入口
- 提供 `getInvalidCells()` 与 `isValid()` 用于合法性检查
- 提供 `clone()`、`toJSON()`、`toString()` 用于复制与外表化

本次改进在 `Sudoku` 内部新增了严格的不变量检查：

- 检查盘面必须为 `9x9`
- 检查单元格值必须是 `0-9` 的整数
- 提示格不可修改
- 非法坐标或非法值输入不会改变状态

### 2.2 `Game` 的职责

`Game` 负责会话管理与历史管理，核心职责如下：

- 持有当前 `Sudoku`
- 管理 `undoStack` 与 `redoStack`
- 提供 `guess()`、`undo()`、`redo()` 作为面向 UI 的操作入口
- 提供 `canUndo()`、`canRedo()` 供 UI 控制按钮状态
- 提供 `toJSON()` 与 `createGameFromJSON()` 实现会话序列化

本次改进重点完善了 `Game` 的会话边界：

- `createGame()` 内部克隆传入的 `Sudoku`
- `getSudoku()` 返回克隆对象，避免外部绕过历史栈直接修改内部状态
- history 反序列化阶段逐条校验 move 结构和值域

## 3. View 层消费方式

### 3.1 直接消费对象

在实现层面，View 层通过 `@sudoku/stores/grid` 访问 Store Adapter 提供的接口，不直接持有可变领域对象。  
Store Adapter 作为唯一桥接点，在内部持有 `Game`，并通过 `Game` 持有 `Sudoku`。

### 3.2 暴露给 UI 的响应式状态

适配层对外暴露如下状态：

- `grid`：初始提示格视图
- `userGrid`：当前可编辑盘面视图
- `invalidCells`：冲突单元格集合
- `canUndoGame`：撤销可用状态
- `canRedoGame`：重做可用状态

其中，UI 只消费用于展示和交互控制的状态，不直接消费领域对象内部引用。

### 3.3 暴露给 UI 的操作命令

适配层对外暴露如下命令：

- `grid.generate(difficulty)`：开始随机新局
- `grid.decodeSencode(sencode)`：从编码加载题面
- `userGrid.set(pos, value)`：用户输入
- `userGrid.applyHint(pos)`：应用提示
- `undoUserMove()` / `redoUserMove()`：撤销与重做

## 4. 真实流程接入说明

### 4.1 开始一局游戏

流程为：生成或解析题面 -> `createSudoku(...)` -> `createGame(...)` -> 同步快照至 store。  
页面渲染的盘面来自领域对象导出的快照，而不是独立维护的旧数组。

### 4.2 界面渲染当前局面

`Board` 组件通过 `$grid` 与 `$userGrid` 渲染盘面，并结合 `$invalidCells` 进行冲突高亮。  
UI 不直接读取或修改领域对象内部引用。

### 4.3 用户输入进入领域对象

键盘输入通过 `userGrid.set(pos, value)` 进入适配层，适配层调用 `game.guess(...)`，再由 `Game` 调用 `Sudoku.guess(...)`。  
落子规则、合法性判定与历史维护全部在领域层完成。

### 4.4 Undo / Redo 接入

撤销与重做按钮分别调用 `undoUserMove()` 与 `redoUserMove()`，内部直接转发到 `Game.undo()` 与 `Game.redo()`。  
历史状态由 `Game` 维护，组件不实现历史逻辑。

### 4.5 领域变化后的 UI 刷新

每次领域操作后，适配层执行同步函数，将 `game.toJSON().sudoku` 的最新快照写入 `writable store`。  
组件通过 `$store` 订阅状态变化，Svelte 自动触发重渲染。

## 5. 响应式机制与边界

### 5.1 使用机制

本方案依赖以下 Svelte 3 机制：

- `writable` / `derived` store
- 组件内 `$store` 自动订阅
- 快照写入触发更新

### 5.2 响应式边界划分

- 领域层（`Sudoku` / `Game`）为业务核心，保持非 UI 依赖
- 适配层为响应式边界，负责领域状态到 UI 状态的映射
- 组件层只消费状态与触发命令，不承载核心业务规则

典型的可见/不可见状态划分如下：

- 对 UI 可见：`grid`、`userGrid`、`invalidCells`、`canUndoGame`、`canRedoGame`
- 对 UI 不可见：`Game` 内部 history 栈、`Sudoku` 内部 `initial/cells` 引用

### 5.3 直接 mutate 的风险

若在组件中直接修改二维数组或直接操作可变领域实例，会带来以下问题：

- 可能绕过 `Game` 的历史管理，造成盘面与 undo/redo 不一致
- 深层对象修改不易被稳定追踪，可能出现状态变化但界面未刷新
- 非法值可能直接写入盘面，破坏领域中的不变量

## 6. 相比 Homework 1 的改进总结

### 6.1 Homework 1 中不足之处

Homework 1 已完成基础对象拆分与功能实现，但还存在以下问题：

- 领域对象和前端主流程衔接不到位，部分状态入口分散在不同层。
- `Game` 与 `Sudoku` 的边界不够稳定，调用方仍有机会绕过统一入口。
- 序列化与历史数据的校验强度偏弱，恢复流程对输入质量依赖较高。

### 6.2 实质性改进项

1. 强化 `Sudoku` 的不变量检查与校验能力。
2. 完善 `Game` 会话边界，阻断外部绕过历史逻辑的路径。
3. 提升统一序列化协议的严格度，补齐版本与 history move 校验。
4. 将前端主流程全部接入 `Game/Sudoku`，移除组件中的核心业务逻辑。

### 6.3 改进后的效果

相比 Homework 1，本次实现在实际使用中的效果更稳定，更容易维护：

- 组件侧主要负责渲染和事件转发，业务逻辑集中在 `Game/Sudoku`。
- 用户输入、撤销和重做都经过统一入口，历史记录与盘面状态保持一致。
- 序列化与反序列化的校验更完整，状态保存与恢复更加可靠。
- 领域层、适配层、视图层之间的职责更清楚，后续扩展成本更低。

### 6.4 设计权衡

本方案在每次操作后都会执行快照同步，存在少量拷贝开销。  
考虑到数独规模固定为 `9x9`，该开销可接受，并换来了更明确的边界、更稳定的更新行为和更高的可解释性。

---

## 7. 总结

本次实现完成了从“领域对象可测试”到“领域对象驱动真实 UI 流程”的转变。  
`Sudoku` 与 `Game` 负责业务规则与状态演进，Store Adapter 负责响应式映射，Svelte 组件负责渲染与交互分发。
